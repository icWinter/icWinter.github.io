后端架构漫谈

1、 什么样的系统架构可以称的上优秀?

    易于功能扩展和维护!
    在进行软件功能的扩展中，对项目源代码零侵入（理想状态下）。或者是尽可能少的影响原代码改动，项目上线也仅仅针对新增功能进行测试。

2、 关于设计原则，以下六种。简单说明下：

    一、单一职责原则
      每个类只负责自己职责内的事情，我个人是这样理解：
        DO模型 ：与数据库严格对应的Bean，只对外提供调用方法，作为一个功能提供者，尽可能的避免去依赖调用的其他模块。
        Service模型 ：本身作为一个提供者与消费者，也是业务逻辑的中心，在系统中应该是最复杂的模块。
      不过我经常在纠结与一个想法，那就是：在一个Service中，偏向于自身方法(或者其他Service方法)的调用，还是偏向于直接进行Mapper的调用？

      就好比上述代码中的 bMethod()与cMethod()的区别,其实我更偏向于cMethod(),因为在Service中的方法尽可能将其进行更加细粒度的划分.对外提供的方法也就会功能更加单一、细化,具体调用方可以自行去根据具体业务来组合这些方法来实现功能.
    二、里氏替换原则
      所有父类出现的地方都可以换成具体子类，并且不会出现任何错误。子类继承父类的时候会自动涵盖父类的功能模块，但是反过来就不可以，子类出现的地方不能用父类去替代。
    三、依赖倒置原则
      面向接口编程，细节依赖抽象。
      接口定义了(method)行为约束，抽象类定义行为和属性约束。所有的实现类都可以向上转型为接口类，调用接口方法。转型为抽象类，获取共有属性，进行抽象类中的方法调用。
      值得一提的是，在jdk1.8中，接口类允许有默认实现。非常有利的就是，如果在设计接口的时候不确定某些方法是否必须要被实现，则可以定义为默认方法。具体实现类可以根据需要进行特定方法重写，有效避免了所有方法重写（记得在之前要实现这种按需重写接口方法的话，需要中间加一个抽象类，在抽象类中重写接口所有方法，然后具体实现类继承该抽象类，接下来就可以按需实现抽象类中的方法。很显然Oracle意识到这点了,哈哈）。
    四、接口隔离原则
      接口的定义也应该符合单一职责原则，尽可能的轻量。避免繁重的接口出现，如有必要可以按照业务类型拆分繁重的接口，毕竟java是允许多实现。
    五、迪米特法则
      每个类尽量减少对其他类的依赖，因此，很容易使得系统的功能模块功能独立，相互之间不存在（或很少有）依赖关系。个人理解就是尽可能的通过已有的类关联实现功能。例如：

    六、开放封闭原则
      软件实体应该对扩展开放，对修改关闭。先说对于修改关闭，大家都可以想到，合理使用final关键字，就可以做到。
但是前提就是在系统设计方面务必考虑周全一点，我曾不止一次遇到过引入的依赖的模块不能完全适用于目前业务需求，导致自己不得不拉取源代码，直接进行源码改动。
所以这种情况就有点类似于：当我们自己写一个工具类的时候，我们并不清楚，该工具类被使用的具体的业务场景。
然而我们还想尽可能的在不修改源代码的基础上，完全兼容不同的业务场景，然而这就引入了扩展开放的概念。
